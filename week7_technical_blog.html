SQL Injection is a way used by Hackers to steal Data from companies that host Web applications. It is a technique used by hackers wherein they insert cleverly constructed SQL Statements into a user input form that will get them user Data like usernames,password,Account no's etc. This poses a serious threat to the Customer's that have their Data stored in the company's Database. Especially in Financial and Banking Web applications such a attack will prove detrimental since the hacker could potentially access the User's Bank Account details! A hacker could also use SQL Injection to wipe out a complete Table.

Let's see some examples of how a hacker can use simple SQL statements to get access to an application's Database or wipe out a Table from the Database:

SQL Injection : 1 = 1 Always True

Select Name,Password from users where user_id = 101 OR 1 = 1

In the above example, 1 = 1 will always evaluate to true, thus giving a list of all Names and passwords in the users table!. A Hacker will just have to input user_id = 101 OR 1 = 1 in the input box and hack the system!

SQL Injections: WHERE ""="" is Always TRUE

Select Name,Password from users where Name = "ABC" OR ""=""

In the above example, WHERE ""="" is always going to be true since "=" = "=", thus giving the hacker access to all the User Data.

SQL Injections: Batched SQL Statements

Select * from users;Drop Table users

In the above example, the hacker can input something like 101;DROP Table users in the input form, which will execute the Statements - Select * from users;DROP Table users against the users table, thus wiping out the entire table!

Prevent SQL Injection:

I have noted below two ways to prevent SQL Injection attacks:

  • Constrain Input: This is as simple as having field validations for length,type,format and range. Having these validations will throw an error when a user enters a SQL statement in the user input field.
  • SQL parameter for Data Access In this case, the user input is passed onto a Stored proecedure that will execute the SQL Statement and the User input. In this case the parameter is treated as a literal value and not as a 'executable code'. The parameter is also checked for its length and type.

These are just a couple of examples to avoid SQL Injection attacks. There are more secure and fullproof ways to avoid SQL Injections.
